①

store 暴露统一的 residency-aware 列举——把虚拟文件系统真正架空

Strong
涉及文件

问题（friction）

store 的卖点是"把文件系统架空成路径寻址的虚拟 FS，离线在线透明"。它对读和去留兑现了这个承诺，唯独 listing 没有：

  • store.listAll() → 纯云端（cloud-sync.ts:308）；store.localKeys() → 纯本地（local-cache.ts:27）。两个半截原语，没有一个"列出这个虚拟 FS 里所有文件"的入口。
  • CloudItem{path,size} 没有 residency 字段——一个文件是"云端/已缓存/脏/local-only"哪种态，列举结果里看不出来。
  • 于是每个消费方必须自己 union：WebPaint 写了 mergeLocalCloud，JRP 新代码忘了 union。
这正是 store 自己禁止的事。 store/CONTEXT.md §反-duplicate（47-57 行）白纸黑字：「本地副本的存在与去留 = store 独占职责……app 层出现任何回答"什么在本地"的逻辑 = duplicate，必删」，leak-test grep 还点名 reconcileCloudGone。mergeLocalCloud 就是这条红线点名的越狱——它只是因为换了名字躲过了 grep。教义用文档禁止，接口却用形状邀请。这就是矛盾，也是漂移的结构性温床。
deletion test

删掉 mergeLocalCloud（和"列本地"那半逻辑）→ 复杂度不会消失，而是在每个消费 app 里重新长出来（WebPaint 长了一次、JRP 该长没长 = bug）。结论：它在挣钱（earning its keep）→ 它该只活一次，活在真相所在地 = store（store 已独占 cloud-sync 的 etag、local-cache 的本地、local-head 的 dirty 三样输入）。

方案（solution）

把列举接口加深成单一 residency-aware 列举：

// store 新面：一次列出整个虚拟 FS，每项自带 residency
type Residency = "cloud-only" | "cached" | "dirty" | "local-only";
interface Entry { path: string; size?: number; residency: Residency; }
store.listEntries(): Promise<{ files: Entry[]; folders: string[]; complete: boolean }>;
// 内部：union(cloud.listAll, local.appKeys) ⋈ local-head(dirty/seenBase)
// 云端那半拿不到（登出/离线/抛错）→ 仍从本地 appKeys 产出 local-only / cached 行
  • localKeys() 降级为内部细节（或保留给批量判，但不再是列举的唯一本地来源）。
  • mergeLocalCloud 删除；WebPaint 回传时一并删（消一处越狱）。
  • offline-first 变成接口的性质，而非 app 的纪律——没有任何消费方能"忘记列本地"。
收益（locality + leverage）
  • locality：union 住在真相所在地。app 永远拿不到 etag/dirty/online——本就喂不到 union 的输入，现在连这个动作都不归它。
  • leverage：消费方一行 listEntries() 拿到离线可用的完整列表；"云端 only 列举"这种错误状态变得不可表达（呼应 store 自己用 local-head.recordEdit 把 bypass 做成不可表达的成功范式）。
  • test surface：union 像 classifyCloudGone 一样成为 store 内的纯函数单测点，而非散在 app 的 try/catch→返空 里（现在那个 catch 把所有失败都吞成空 gallery，测都没法测）。

问题：太监的本质

"太监"= 一个本该被听见、却生在最深处局部副作用里的身份事件。具体链条（实证）：

  • doc-id.ts：contentDocId = 整块 PDF 字节 SHA-256 → c-<16hex>，当 catalog item 的 key。
  • catalog.ts：catalog 完全 key by docId，fileName 只是可变 payload → 这是第二身份（store 身份=path/name）。
  • openPaper（app.ts:222-226）每次开都只从字节算 docId，分支全静默： 
    const docId = await contentDocId(await blob.arrayBuffer());   // 只认字节
if (!cat.get(docId)) cat.upsert(docId, {fileName: item.name, ...}); // (b) 新生
else              cat.upsert(docId, {fileName: item.name});          // (a)(c) 静默改绑
case发生什么太监在哪
(a) 同内容两路径同 docId → fileName 被静默改成"最后打开的那个" → 两份文件抢一个阅读位置没人知道两个文件撞成一个身份
(b) 内容变（损坏/截断）新 docId → 新 entry；老 docId entry 永不回收，成幽灵行，老位置永远够不到字面狸猫换太子；孤儿 entry 无任何代码会清
(c) 外部移动/改名下次开 → 同哈希 → fileName 静默 re-bind 到新路径是身份事件，但没人听见
身份逻辑摊在 app.ts 各 handler inline（openPaper / onGalMove / onGalRename / onGalTrash / jumpscare），没有一个"解析这个文件的身份"模块。case (b) 的孤儿根本没有家。这违反 store 自己的 Model-B「决策必 surface」——只不过违反者是 app，不是 store。
hash 去重该不该是 store 的事？（你的直觉 vs 红线）

直觉合理，但对通用 store 不成立：store 格式盲（MASTER §A：identity=path/name，GUID-in-content 真机后 rollback），而编辑器的内容可变——每改一笔哈希就变，WebPaint 这种 app 用不了内容哈希身份。所以 hash 去重不进 cloud-store。但对只读不可变内容（论文/书/音频）它是合法身份——因此它该住在 store 的兄弟库（读者域），给 JRP/JustReadBooks/音频位置复用。这正是它泄漏进 app 的原因：它需要一个家，而家一直没建，于是落在了最近的地面=openPaper。

方案

先做 docs/13 的先决决策（A 砍 / B 留），再动代码：

  • 选项 A（更简）：砍内容哈希，catalog 改 key by 路径 → 单 SSoT、零静默裁决。代价：外部 OneDrive 移动后位置重置（= wart E 接受到位置上）。
  • 选项 B（更强）：建读者域模块 library，open(path) 返 {blob, docId, position, event}，event ∈ {born, recognized, moved, content-changed-orphan}；openPaper 退回纯 UI；孤儿回收有家；裁决按 Model-B surface。

为什么 AI 会漂移（这才是你真正在问的）

离线 bug、太监、mergeLocalCloud 越狱——表面是三个独立故障，根子是同一个机理。

1 · 浅/泄漏的接口逼"重新推导"，重新推导就是漂移的入口

listAll+localKeys 两个半截，逼每个 app 重推 union。重推一次，就可能漏一个分支——JRP 漏了本地。而 open() 那种深接口（一行拿到本地或云）没人能漏。深接口让错误不可表达；浅接口把正确性外包给纪律，纪律必腐。

2 · 文档强制的不变量必漂；类型强制的不变量不漂

CONTEXT.md 禁止 app 碰"什么在本地"——但接口形状要求它碰。编译器抓不到这个矛盾。mergeLocalCloud 还骗过了 leak-test grep（换了名字）。tripwire（grep）不是结构屏障。 把不变量挪进类型/接口（候选①）才治本——这正是 store 用 recordEdit 把 bypass 做成"不可表示"的同一招。

3 · 移植 = 从浅接口重推导

JRP 是从 WebPaint port 过来的。port 不是复制行为，是对着浅接口重新实现。WebPaint 的 offline-first 住在 app 层的 mergeLocalCloud 里——port 时只搬了 store、没搬那段 app 越狱逻辑，编译器对"云端 only 列举"很满意。能力就这么蒸发了。

4 · doc 是没有失效机制的缓存，会在代码漂移之上再漂一层

handoff P1 把这个回归误标成"不是回归、是新增能力"——代码漂移之上叠了一层文档漂移，差点让下一个 agent 当 feature 去做。信任顺序（CLAUDE.md）：代码现状 > 人类原话 > ADR > AI 写的 how-doc。本报告的对照表就是用代码现状纠了那条 doc。

统一教训：太监（身份没有家→落在 openPaper）和离线 bug（residency 没有家→app 漏拼）是同一个病——一个职责没有结构性的家，就会落在最近的地面，并在每次重推导时悄悄变形。治法不是更多文档/grep/审查，而是给它一个深模块的家，让错误的形状无法编译。store 早就证明过这招有效（local-head 的 bypass-不可表示）；候选①②就是把同一招用到 listing 和 identity 上。

Top recommendation — 先动哪个

先做 候选① — store 统一 residency-aware 列举

它是离线回归的结构性根因，同时一举溶解 mergeLocalCloud 这一整类越狱（WebPaint 回传时一并删）。改完后，"忘记列本地"在每个 sibling 里都不可表达——这比修 JRP 一个 app 的三层 gating 收益大一个数量级，因为它治的是会在每次 port 复发的病根。

路径：触 store 列举接口 = 红线区 → 走 pwa-cloud-store skill + escalate（改前确认 DATA SAFETY GUIDELINE.md）。不发明红线，只把 CONTEXT.md 的承诺从文档变接口。

之后：②（读者域库收太监，先定 docs/13 的 A/B）与 ③（folder-tree 共享 UI）同批做（handoff P2+P3）。顺序的理由：③ 的输入正是 ① 的列举 + ② 的身份事件，所以 ①→②→③ 是自然的依赖序。

临时止血（若想先让用户能离线看、再做深化）：在 persistence/index.ts:listGallery 把 store.localKeys() 并进 files（catch 云端失败时保留本地行）+ 删 app.ts:64 的 !signedIn 硬清空 + 改 gallery.ts:223 空态文案。但这是 app 层补丁，会再造一处 mergeLocalCloud 同类越狱——只当过渡，终态以候选①为准。
