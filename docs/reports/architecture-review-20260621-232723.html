每个红线职责成独立深模块（小接口、大行为），吃脊椎（substrate + base-lineage）+ cloud/local。createStore 只剩接线 + 新 API facets。每块可单测。

⚠ 两条 etag 轨道，拆分时绝不能并

cloud-sync 拥 kv 里的持久/跨 tab etag+dirty；store.ts 另拥一个 _base（本 tab 已见云版）。 push 的 If-Match 用 _base/_parent，绝不用 kv 共享 etag——否则别的 tab 推成功改了共享 etag，本 tab 陈旧推被误判"无冲突"→ 静默覆盖（W2）。 base-lineage 拥 _base，cloud-sync 拥 kv，两者只在 open/adopt 时单向 seed。

深化候选

候选 1 · 脊椎

抽出 base-lineage — 每-tab 云版谱系 + bypass 守卫

Strong
涉及

_base · _parent · seenBase · captureParent · reparent · clearParent · parentFor · hasParent · adoptBase · cloudState.setDirty 门 · _doPush 的 bypass 守卫

问题

整个库最危险的红线（W2：push 必须以 parentBase 为 If-Match，绝不用共享 etag，否则静默覆盖）现在是 9 个闭包函数 + 2 个 Map，被 5 个职责（push/freshness/delete/identity/seal）读。没有一处叫"这个 tab 的未推编辑派生自哪个云版"。bypass 守卫（那个宁可抛错也不静默覆盖的判断）零测试。

删除测试

把它内联掉 → 谱系逻辑在每个会 push/adopt 的 flow 里重新长出来（5+ 处）。复杂度集中重现 = 它在挣自己的工钱 → 该成深模块。

方案

一个有状态 base-lineage 模块拥 _base+_parent，暴露小接口：ifMatchFor(name) / seenBase / capture/reparent/clear / adopt(name,etag) / assertNotBypassing(name)。一个实例穿过所有 flow。

收益（locality + leverage + 测试）
  • Locality：最致命的不变量住一个文件、一套测试。
  • Leverage：push/freshness/delete 从 3 方法接口拿到 If-Match 正确性，不再各自重推。
  • 测试：mock-provider 的 fault-inject 已就绪 → 注入陈旧 base → 断言抛错而非覆盖。

候选 2 · 红线

抽出 safe-resolve — 永不丢字节（备份/校验/自愈/化解）

Strong
涉及

_safePull · _tryHeal · _resolveConflict

问题

A4/A10（备份先于覆盖）、R1（采纳后置 etag）、N2（validateAdopt 闸挡 captive-portal HTML）、B5（lost-response 自愈）四条红线是 3 个私有函数，同时被 push 和 open/freshness 调。最灾难的"备份→校验→覆盖→采纳"顺序没有任何测试。

删除测试

内联 → push 和 freshness 各自重写"备份再覆盖"。复杂度集中重现 → 深模块。

方案

safe-resolve 模块：safePull(name, adopt) / tryHeal(name, bytes) / resolveConflict(name, choice, ctx)。拥那条不可重排的顺序；吃 cloud(pull/weakOverride) + base-lineage(落地后 commit etag) + local(backup/save)。

收益
  • 永不丢的顺序集中一处。
  • 测试现成：mock-provider 的 lostResponse fault 已建 → 断言 heal 收敛、断言 dirty 先备份后覆盖。

候选 3 · 红线 · 最高测试欠债

抽出 seal — at-rest 加密透明

Strong
涉及

_seal · _unseal · _unsealOrThrow · _withPassword · _lockedErr · verifyPassword · getTailBytes · decryptPeekBytes · readPeek · crypto-container 全部耦合

问题

加密透明（encode 永出明文、写前包壳读后解壳、非交互密码）摊在 ~12 个函数，且泄漏进 open(:206 looksEncrypted) 和 safe-resolve(:293 weakOverride encrypted 标志)。crypto-container 的 _codec/_keyCache 是模块级单例。JRP 里零加密测试——全库最大的未测红线。

删除测试

内联 → 每条读/写路径重写"是不是加密？解/包壳"。集中重现 → 深模块。

方案

seal 模块包住 crypto-container：sealForWrite(name, plain) / unsealForRead(name, bytes) / withPassword / verifyPassword / readPeek / isContainer。加密知识收一处；open/safe-resolve 经 seal.isContainer() 这个 seam 调，不再直 import crypto-container。

收益 + 待解子问题
  • 唯一懂容器的地方；可测（seal/unseal round-trip、locked→status、错密码）。
  • 对接 STORE.md 的 sevenZip 注入（#78）+ ZipFile.setPreview。
  • ⚠ 子问题：_codec 模块单例 → seal 要每-实例需先动 configureCryptoCodec。grill 时定。
候选 4 · flow 群

把 push / freshness / delete / identity / trash 提到脊椎上

Worth exploring
问题

这 5 个是各自独立的单一职责 flow，现在共用一个闭包 → 看似耦合。实际每个只碰脊椎 + cloud/local + 自己的 opts。耦合是偶然的（共享闭包），非本质。delete 的 base-guard 重放（"旧设备苏醒，绝不删别人的同名新文件"）这条红线也没测。

删除测试

把任一 flow 内联回 → 其它 flow 不受影响（它们本就不互调，除 identity 内部走 _doPush）。说明它们本是独立模块。

方案

每个 flow 成模块，吃 (substrate, base-lineage, safe-resolve, seal, cloud, local) + 自己 opts。createStore 接线。先做完候选 1-3（脊椎/safe-resolve/seal）再做这层。

收益
  • 每个 flow 对着 mock-provider 单测。
  • delete-queue 的 base-guard 重放拿到专测。
  • 对接新 API：push/freshness 喂 RawFile.save/open，delete/identity 喂 file.delete/rename。
候选 5 · 组合根 + 新 API

createStore 收成薄组合根 + STORE.md facets

Worth exploring
问题

今天 createStore 就是那 1050 行——既是组合根又是每个实现。新 STORE.md API（file/collection/settings/encryption + Model B ui bundle）需要一个是胶水、不是 guts 的落脚处。

删除测试（反向）

候选 1-4 抽走后，createStore 删掉的复杂度不重现在别处——剩下的纯是接线（provider→cloud/local、实例化模块、映射 ui）。证明它本该是薄的。

方案

createStore 缩到 ~150 行接线：build cloud/local from provider → 实例化脊椎 + flow 模块 → 把 ui={busy,askPassword,resolveConflict,reportError} 映射到各 flow 回调 → 暴露 file()/collection()/localSettings/syncedSettings/encryption。

收益
  • 组合根读起来像接线图。
  • 新 API 面与红线 guts 解耦；ui 映射住一处。
  • 胶水（busy/autosave/single-flight/edit）留这里，薄。

首选：从脊椎切第一刀

按 候选 1 → 2 → 3 → 4 → 5 的依赖序走，每抽一个模块当场补它的单测（把红线落进测试，给这次重构兜底）：

  1. 1
    base-lineage 先抽——它是 W2 脊椎，其他全依赖它；先把 bypass 守卫（最危险、零测的红线）落进测试。
  2. 2
    safe-resolve + seal——两条永不丢/加密红线，都没测，且 mock-provider 的 fault-inject 已就绪，测试成本低。
  3. 3
    flow 群 逐个提到脊椎上，各自单测。
  4. 4
    createStore 收成薄组合根 + 接 file/collection/settings/encryption + ui 映射。

这条路符合家族"从中间切蛋糕、禁止刮痧"——职责线就是深切线；guts 逻辑一行不改，只是搬家到各自文件 + 补测。 且不违反"同步引擎改前 escalate"（本报告即 escalation；guts 不动）。collection / settings 已照此抽好、测过（42 passed）——是这套打法的样板。
