① pin / evict —— 「AI 是不是凭空造了 pin 逻辑？」

不是漂移 · JRP 领先

store/pin-set.ts · store/evict.ts · store/create-store.ts:80-142 · vs WebPaint/src/store/*（无） · WebPaint/docs/20260528-backlog.md:640 (N4)

现象 / 你的疑问

MASTER §A 把「驱逐红线（dirty 永不被驱逐、auto-pinned）」列为核心。你以为这套 pin/evict 是 WebPaint store 的红线中心，却看到 JRP store 里冒出全新的 pin-set.ts + evict.ts，于是怀疑：要么 WebPaint store 有根本缺陷，要么 JRP 的 agent 在编造。

查证（代码现状）

  • WebPaint src/store/ 里 pin/evict 命中数 = 0。 全 repo 的 pin 命中全是 UI 取色器气泡（color-panel.ts 的「吸色 pin tooltip」），与存储无关。
  • WebPaint 的驱逐守卫不在深模块——它躺在 app 层 session-state.ts:492-502，而且是已知债：backlog N4「驱逐 cloud-dirty 副本无 .backup stash，guard 不在深模块」。
  • WebPaint 的 handoff doc 把 pin 列为「app owns 的云表面 7 项」之一——计划要做、从未抽成深模块。
  • JRP 的 evict.ts 把整条红线收进一处：只驱逐 clean ∧ 在线 ∧ 云端仍有 ∧ !pinned，自述「逻辑零新红线＝组合 head.isDirty / cloud.fetchMeta / local.exists/hardDelete + pin-set」。
  • JRP 干净地分开了两种 pin（见下图），且没有第二真相源：pin 态只活在 pin-set(kv)，cached 态只活在 local.exists；gallery-model / content.ts 只是注入谓词函数去读，不自管。

判定

两个选项都不完全对，准确说法偏第一个：WebPaint store 不是「有根本缺陷」，而是「不完整」——驱逐红线从没被抽进 store，一直以 app 层债（N4）形式存在。JRP 的 agent 没有编造：它实现了 MASTER §A 一直要求、WebPaint 一直没写的那条红线，做成了教科书式单漏斗深模块；而 pin-set（per-device 离线常驻）是 JRP 产品真需要的新功能（「各端接着读、飞机上能看」），WebPaint 这种「本地全留」的画板 app 从不需要。这正是你说的「我在 JRP 精炼 interface 与 scope」。

用「不重推红线」规则复核（碰 pin/evict/缓存/加密/冲突 绝不手搓重导，找旧 vetted 版对齐移植，drift=毒）

这条规则正是为 pin/evict 这种场景设的。复核 evict.ts：它没有手搓新红线——自述「零新红线＝组合 head.isDirty/cloud.fetchMeta/local.exists/hardDelete + pin-set」，并显式对齐到设计文档（注释引 state-machine §1-2、potential-bugs I6/J6、MASTER §A）。规则允许的 vetted 源包含「设计文档」，所以这是合规的「对齐移植」而非「手搓重导」。
唯一需要额外验的：WebPaint 没有代码层祖先可 port（它的守卫在 app 层 session-state.ts:492-502 且带 N4 bug），所以 evict.ts 是按设计文档新写的红线代码。建议拿它对 MASTER §A + WebPaint 那段 app 守卫逐条做 parity diff 确认无静默偏差，并把它定为共享 vetted 版（见 ⑥ 反向传播）。

② jailbreak：app 层 deep-import 进 store 内部

真违规 · 但极轻（type-only）

src/persistence/content.ts:5

Problem

import type { Bytes } from "../store/types.ts"; —— 这是从 store 内部文件直接拿类型，绕过了唯一入口 index.ts。types.ts 顶部写着「store 深模块共享类型契约」，且 index.ts 没有 re-export Bytes。这正是 STORE.md 铁律 #4 + build lint 要挡的 deep-import：它绕过了「app 只跟一个入口打交道」这个漏斗。

为什么仍然要修（即便它「只是个类型」）

Bytes 不过是 Uint8Array，import type 运行期被擦除、碰不到任何 guts，所以当下零数据风险。但漏斗的价值在「没有例外」——一旦默许一个「无害的」deep-import，lint 白名单就开了缝，下一个 agent 会照例外推进到运行期 import。正解：escalate，让 index.ts re-export Bytes（库 API 缺口，按铁律 #3 改库、不在 app 端绕）。

Before

  ├─ import {…} from "../store/index.ts"   ✓
  └─ import type {Bytes}
        from "../store/types.ts"           ✗ deep
After
index.ts  export type {Bytes}
content.ts
  └─ import type {Bytes}
        from "../store/index.ts"           ✓ 漏斗

其余 app 层（ui/、domain/、config、app-state、catalog、gallery-model）全清白：无 localStorage / indexedDB / MSAL / Graph / 裸 fetch；唯一碰 store 的是 persistence/，且只走 index.ts（这一处例外）。config.ts 的 MSAL 字符串是喂给 createOneDriveProvider 的惰性常量，非违规。
